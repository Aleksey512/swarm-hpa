# CI golangci-lint step failed: action@v6 installs v1, incompatible with v2 config and go1.25

**Date:** 2026-07-01 13:44
**Files:** .github/workflows/ci.yml
**Severity:** high

## Problem

The `build-test` job's `Lint (golangci-lint v2)` step failed on every push since
CI was introduced (`golangci-lint exit with code 3`). The runner log showed:

```
Error: can't load config: the Go language version (go1.24) used to build
golangci-lint is lower than the targeted Go version (1.25.0)
```

`golangci/golangci-lint-action@v6` with `version: latest` installed
golangci-lint **v1.64.8** (built with go1.24), while `.golangci.yml` is a
**v2-schema** config (`version: "2"`) and `go.mod` targets `go 1.25.0`.

## Root Cause

Two coupled mismatches, both rooted in the action major version:

1. **Action ↔ linter major version.** `golangci-lint-action@v6` only installs
   golangci-lint **v1.x** — even `version: latest` resolves to the latest v1
   (1.64.x). golangci-lint **v2** requires the **v7** action. The `.golangci.yml`
   is v2 schema, so a v1 binary cannot read it.
2. **Linter build Go version < module target.** golangci-lint refuses to run
   when the Go toolchain it was built with (go1.24 for v1.64.8) is older than the
   module's targeted Go version (`go 1.25.0`). This is a hard error (exit 3),
   independent of any lint findings — so the step never even evaluated the code.

The code itself was clean: local golangci-lint v2.12.2 (built go1.26.4) reports
`0 issues`. The failure was purely a CI toolchain-version bug, not a code defect.
It went unnoticed because local `make lint` used a correct v2 install while CI
silently used a v1 install via the older action.

## Solution

In `.github/workflows/ci.yml`, bumped the lint step:

```diff
- uses: golangci/golangci-lint-action@v6
+ uses: golangci/golangci-lint-action@v7
  with:
-   version: latest
+   version: v2.12.2
```

`@v7` is the v2-capable action; pinning an explicit v2 release (built with
go ≥ 1.25) satisfies both the v2 config schema and the go1.25 target, and avoids
`latest` ever resolving to an incompatible build again.

## Prevention

- **Match the action major version to the golangci-lint major version:** v6→v1,
  v7→v2. When `.golangci.yml` declares `version: "2"`, CI must use
  `golangci-lint-action@v7`.
- **Pin the linter version in CI** rather than `version: latest` — an unpinned
  linter can silently drift to a build whose Go version is older than `go.mod`'s
  target and hard-fail (exit 3).
- **Keep the CI linter's build-Go version ≥ `go.mod` target.** When bumping the
  `go` directive in `go.mod`, verify the pinned golangci-lint release is built
  with a Go version at least that high.
- **Watch CI after the run that introduces it** — this step was red from its
  first run; a quick `gh run list` after adding a workflow catches
  "green-locally, red-in-CI" toolchain gaps immediately.
